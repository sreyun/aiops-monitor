仓库采用单 Go module `aiops-monitor`，通过两个 `cmd/*` 入口编译出独立二进制：
- `cmd/server/main.go`：HTTP API 服务，统一依赖 PostgreSQL（关系数据）和 VictoriaMetrics（时序指标），通过环境变量 `AIOPS_POSTGRES_DSN` / `AIOPS_VM_URL` 注入；内置 gzip 压缩、CORS、安全头、TLS 可选、优雅关停等中间件链。
- `cmd/agent/main.go`：主机侧守护进程，支持多服务端广播上报（collect once, broadcast all）、Relay 网关模式、Python 插件执行、日志收集转发，并新增三类采集器：Redfish（BMC/iDRAC/iLO 硬件状态）、NetFlow v5/v9 UDP 接收聚合、Linux nf_conntrack 五元组增量采集。
- `shared/wire.go`：定义 Agent↔Server 共享的 wire types（Report/Metrics/HardwareSnapshot/NetFlowReport/FlowRecord 等），保证两端结构体契约不漂移。

数据流分层：
1) 基础指标：Agent 原生 collector → POST `/api/v1/agent/report` → Server 落 PG + 写 VM。
2) 硬件快照：`collector_redfish.go` 周期轮询 BMC → `postHardwareReport` → `/api/v1/agent/hardware` → PG upsert + vmHardwareMetrics → VM。
3) 流量明细：`collector_netflow.go`/`collector_packet.go` 按 window 聚合为 `NetFlowReport` → `/api/v1/agent/netflow` → PG 存明细 + VM 写 bytes/packets 指标。
4) 前端查询：`hardware_netflow.go` 暴露健康历史（VM PromQL）、Top-N 聚合（VM）、明细列表（PG）等 REST 接口。

可靠性设计：每个 serverTarget 独立连接池 + 指数退避 + 熔断器（8 次失败/15s 冷却），并在断路器打开时重置注册态以应对重启；HTTP/2 显式禁用以避免单连接故障导致全量请求失败。
