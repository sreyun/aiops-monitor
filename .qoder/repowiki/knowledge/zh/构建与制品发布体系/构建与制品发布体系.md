## 1. 构建系统概览

本项目采用 **多语言、多产物** 的构建体系：Go（服务端 + Agent）、Android/Kotlin（移动端）、Python（插件层），通过 GitHub Actions 驱动 CI/CD，Docker Buildx 完成多架构镜像构建，最终产出二进制、容器镜像和可分发安装包。

- **Go 模块**：`go.mod` 声明 `module aiops-monitor`，Go 1.22；依赖使用 `-mod=vendor` 模式，`vendor/` 目录提交到仓库，保证离线构建一致性。
- **版本注入**：所有 Go 二进制均通过 `-ldflags "-X main.appVersion=${VERSION}"` 注入版本号，版本来源为 Git tag（`v*`）或 commit short hash（dev 分支）。
- **交叉编译**：本地 PowerShell 脚本 `build.ps1` 支持 Windows/Linux/macOS 三平台交叉编译；CI 使用 Docker Buildx 在 `linux/amd64` 和 `linux/arm64` 上并行构建。

## 3. 构建流程与约定

### 3.1 本地开发构建
```bash
# Windows 原生构建
powershell -File build.ps1

# 交叉编译 Linux/macOS
powershell -File build.ps1 -CrossCompile
```
输出至 `bin/` 目录：`aiops-server.exe`、`aiops-agent.exe`，以及对应平台的非 Windows 变体。

### 3.2 容器镜像构建
```bash
# 本地开发镜像
docker compose up --build

# 生产镜像（指定版本）
docker build --build-arg VERSION=v5.5.0 -t aiops-server:v5.5.0 -f docker/Dockerfile .
```
Dockerfile 使用多阶段构建：
- **builder 阶段**：在 `BUILDPLATFORM` 上以 `CGO_ENABLED=0` 交叉编译，注入 `main.appVersion`
- **server 阶段**：Alpine 基础镜像，仅包含 `aiops-server` 二进制 + 预打包的 `dist/`（含全平台 agent 二进制、配置示例、plugins.zip）
- **agent 阶段**：Alpine + Python3，安装 `psutil`，运行 `aiops-agent`

### 3.3 CI/CD 发布流程
- **触发条件**：推送 `v*` 标签（如 `v5.5.0`）或手动触发 workflow_dispatch
- **版本解析**：从 `GITHUB_REF_NAME` 提取 tag，作为 `VERSION` 注入 ldflags
- **多架构构建**：`linux/amd64, linux/arm64` 双架构，使用 QEMU 模拟 arm64
- **镜像仓库**：华为云 SWR（`swr.cn-east-3.myhuaweicloud.com/sreyun/`），自动登录（AK/SK HMAC-SHA256）
- **产物标签**：`aiops-server:<version>` + `aiops-agent:<version>`，可选 `:latest`

### 3.4 安全门禁流水线
每次 push/PR 自动执行：
- **硬门槛**：`go vet ./...`、`go test ./...`、`govulncheck`、`gitleaks`（全历史扫描）
- **报告级**：`gosec`（SARIF 上传 Security 页）、`staticcheck`、`race` 竞态检测
- **SBOM**：CycloneDX JSON 格式，供企业审计

### 3.5 远程部署
`deploy.sh` 提供一键部署：SCP 上传新二进制 → systemctl/pkill 停止旧服务 → 原子替换 → 重启验证。

## 4. 设计决策与约束

- **无 Makefile**：构建逻辑集中在 `build.ps1` 和 `Dockerfile`，Windows 优先（PowerShell），Linux 通过 Docker 覆盖。
- **Vendor 模式**：强制使用 `GOFLAGS=-mod=vendor`，禁止网络拉取依赖，确保构建可重复。
- **CGO 关闭**：所有 Go 二进制均以 `CGO_ENABLED=0` 构建，避免动态链接库差异导致跨平台问题。
- **版本即 tag**：版本号严格来自 Git tag，未打 tag 时使用 `dev-<commit>` 前缀，便于追溯。
- **多产物打包**：单个 server 镜像内嵌所有平台 agent 二进制，通过 `/dl/` API 按需下发，简化 Agent 安装。
- **插件独立依赖**：Python 插件通过 `pip3 install --no-cache-dir --break-system-packages psutil` 在镜像构建时安装，不污染宿主环境。

## 5. 开发者应遵循的规则

1. **新增 Go 依赖后必须更新 vendor**：`go mod tidy && go mod vendor`，否则 CI 会失败。
2. **修改 Dockerfile 需验证多架构构建**：本地可用 `docker buildx build --platform linux/amd64,linux/arm64` 测试。
3. **版本号管理**：发布前打 `vMAJOR.MINOR.PATCH` 标签，CI 会自动构建并推送镜像。
4. **敏感信息不入仓**：`deploy.sh` 中的服务器地址、密码等应通过环境变量或密钥管理工具注入。
5. **Android 构建**：需在 Android Studio 中打开 `android/` 目录，Gradle 版本由根 `build.gradle` 统一管理。
