## 1. 构建系统概览

本项目采用 Go 单模块 + Docker 多阶段镜像 + GitHub Actions CI 的混合构建方案：
- Go 源码位于 cmd/server、cmd/agent，通过 go build -ldflags "-X main.appVersion=..." 注入版本号；
- 本地开发使用 PowerShell 脚本 build.ps1（Windows）或 docker/Dockerfile（跨平台）；
- 生产制品以 Docker 镜像形式推送至华为云 SWR，同时提供二进制分发包供裸机部署。

## 2. 关键文件与职责

- go.mod / go.sum：Go 模块声明，依赖仅含 lib/pq、ledongthuc/pdf、skip2/go-qrcode 三个第三方库，保持极简
- docker/Dockerfile：多阶段构建。builder 阶段用 CGO_ENABLED=0 原生交叉编译 server/agent 二进制，并额外产出所有平台的 agent 安装包；server/agent 两个最终镜像基于 Alpine 3.20，分别嵌入前端静态资源与 Python 插件
- .github/workflows/release.yml：触发条件为 push v* 标签或手动 workflow_dispatch；使用 setup-buildx + QEMU 构建 linux/amd64,linux/arm64 双架构镜像，登录华为云 SWR 后推送到 sreyun/aiops-server 与 sreyun/aiops-agent 仓库，支持 :latest 同步更新
- build.ps1：Windows 本地一键构建，自动从 Git tag 解析版本并通过 -ldflags 注入 main.appVersion，可选 -CrossCompile 参数生成 Linux/macOS 二进制
- deploy.sh：面向裸机部署的辅助脚本：SCP 上传 aiops-server.exe → 停止旧进程 → 替换二进制 → 启动新服务（兼容 systemd 与非 systemd 环境）
- docker-compose.yml：编排 aiops-server + VictoriaMetrics + PostgreSQL 三容器，默认拉取 SWR 预构建镜像，也支持 --build 本地构建；提供健康检查、端口映射、数据卷挂载等完整运行配置

## 3. 架构与约定

### 3.1 版本注入策略
- 版本号来源优先级：Git tag > commit short hash > dev
- 统一通过 -ldflags "-X main.appVersion=<version>" 注入到 main.appVersion 变量，Server/Agent 均读取同一入口
- CI 中由 GITHUB_REF_NAME 或 workflow_dispatch 输入参数决定，确保镜像内嵌版本与发布标签一致

### 3.2 交叉编译与多架构
- Dockerfile 在 builder 阶段利用 TARGETOS/TARGETARCH 原生编译当前目标架构的二进制
- 同时显式遍历 linux/amd64, linux/arm64, darwin/amd64, darwin/arm64, windows/amd64 构建全部平台 agent 安装包，随镜像一起分发，供 Server 的 /dl/ 下载端点供远端 Agent 安装
- CI 通过 docker/setup-qemu-action 启用 arm64 模拟，实现 amd64 Runner 上构建 arm64 镜像

### 3.3 依赖管理
- 使用 Go Modules，未启用 vendor 目录（Dockerfile 中 COPY vendor/ 仅为占位，实际走网络拉取）
- 依赖极少且稳定，无动态 CGO，保证 CGO_ENABLED=0 下可完全静态链接，产物可直接复制到任意同架构 Linux/macOS/Windows 主机运行

### 3.4 制品形态
- Docker 镜像 (aiops-server, aiops-agent)：华为云 SWR sreyun/*，容器化部署推荐方式
- 二进制包 (bin/aiops-server-*, bin/aiops-agent-*)：本地 bin/ 目录，裸机直接运行或二次打包
- 安装包 (dist/plugins.zip + 各平台 agent)：镜像内 /app/dist/，Server 启动时内置，供远端 Agent 自动下载安装

## 4. 开发者应遵循的规则

1. 新增依赖：仅在 go.mod 中添加，避免引入 CGO 依赖；若必须使用，需在 Dockerfile 和 CI 中调整构建环境
2. 版本号：不要硬编码版本号，一律通过 -ldflags "-X main.appVersion=..." 注入；本地调试可用 git describe --tags 获取
3. 跨平台构建：优先使用 docker/Dockerfile 的 multi-stage 模式，确保 CGO_ENABLED=0；如需纯本地快速构建，使用 build.ps1
4. 发布流程：打 v* 标签后 CI 自动构建并推送镜像；如需补推，可通过 Actions 页面手动触发 workflow_dispatch 指定 tag
5. 本地开发：推荐使用 docker compose up -d --build 一键拉起 Server + VM + PG 全栈环境，无需手动编译
6. 安全注意：Dockerfile 注释已指出 Go 1.22 存在已知 CVE，发布前需升级至受支持的 Go 版本并同步基础镜像
