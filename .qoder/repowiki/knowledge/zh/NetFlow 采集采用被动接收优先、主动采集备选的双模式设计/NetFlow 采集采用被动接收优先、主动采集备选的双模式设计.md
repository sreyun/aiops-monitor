# NetFlow 采集采用被动接收优先、主动采集备选的双模式设计

_来源：4ba2fed → 27b8c5e 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
网络设备（防火墙/交换机）获取流量统计有两种方式：设备主动推送 NetFlow 到 Agent，或 Agent 轮询设备 API。不同厂商和设备类型支持情况差异很大。

## 决策驱动
- 部署灵活性（无需改设备配置）
- 实时性要求
- 实现复杂度控制
- P0/P1 分阶段交付

## 备选方案
- **仅主动采集（SNMP/REST 轮询）** _（已否决）_ — 优点：实现简单，只需一个客户端；缺点：轮询间隔内数据丢失；对设备有额外压力；不同厂商 API 适配工作量大
- **仅被动接收（UDP 监听 v5/v9）** _（已否决）_ — 优点：实时性好，零侵入设备；缺点：需要网络设备提前配置推送地址；某些老旧设备不支持 NetFlow
- **被动接收 P0 + 主动采集 P1 渐进式** — 优点：P0 快速上线通用场景；P1 覆盖更多设备；两种模式输出统一 FlowRecord 结构；缺点：需要维护两套采集逻辑；ActiveTargets 配置增加复杂度

## 决策
P0 实现 UDP 被动接收（v5/v9 解析 + flowAggregator 内存聚合窗口），P1 扩展 SNMP/REST 主动采集；两种模式都产出统一的 FlowRecord 结构体，通过 source 字段区分数据来源。

## 影响
flowAggregator 需要支持滑动窗口、内存上限（100K flows）、背压丢弃计数；但实现了平滑演进路径，先解决最常见的被动接收场景。
