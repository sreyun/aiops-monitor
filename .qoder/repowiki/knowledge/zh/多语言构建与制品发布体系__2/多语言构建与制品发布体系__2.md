## 1. 系统概览
本项目采用「Go + Docker + GitHub Actions」的多产物构建体系，同时覆盖服务端、Agent、Android 客户端三类二进制/镜像制品。核心思路是：以 Go module + vendor 为依赖管理基础，通过 Dockerfile 多阶段构建实现跨平台交叉编译，再由 GitHub Actions 在打 tag 时自动产出 linux/amd64 + linux/arm64 双架构镜像并推送至华为云 SWR；本地开发则提供 PowerShell 脚本一键构建 Windows 二进制。

## 2. 关键文件与职责
- `go.mod` / `go.sum` — Go 模块声明（Go 1.22），配合 `vendor/` 目录锁定依赖版本，Docker 构建使用 `-mod=vendor` 离线编译。
- `docker/Dockerfile` — 生产级多阶段镜像定义，包含 server 与 agent 两个 target，并在 builder 阶段一次性交叉编译所有 Agent 平台二进制（linux-amd64/arm64、darwin-amd64/arm64、windows-amd64）放入 `/out/dist/`，供 server 的 `/dl/` 端点分发。
- `docker/Dockerfile.dev` — 开发用精简版，跳过 dist 交叉编译，仅构建当前目标平台，加速 `docker compose up --build`。
- `docker-compose.yml` — 编排 aiops-server、victoriametrics、postgres、aiops-agent 四服务，默认使用华为云 SWR 镜像，支持本地 `--build` 从源码构建。
- `.github/workflows/release.yml` — Release 流水线：push v* tag 或手动触发 → QEMU + buildx 多架构构建 → HMAC-SHA256 生成 SWR 登录凭据 → 推送 `aiops-server` 与 `aiops-agent` 镜像到 `swr.cn-east-3.myhuaweicloud.com/sreyun`，同时可选打 `:latest`。
- `.github/workflows/codeql.yml` — 每周一定时 + PR 触发 CodeQL SAST 分析（Go），使用 Go 1.26 官方镜像规避 stdlib CVE。
- `build.ps1` — Windows 本地构建脚本，读取最新 git tag 注入 `-X main.appVersion`，输出 `bin/aiops-server.exe`、`bin/aiops-agent.exe`，支持 `-CrossCompile` 参数产出 Linux/macOS 二进制。
- `deploy.sh` — 简易 SSH 部署脚本，将预编译的 `aiops-server.exe` 上传到远端主机并通过 systemctl/nohup 重启服务。
- `android/build.gradle` + `android/app/build.gradle` — Android 应用 Gradle 构建（AGP 8.5.2、Kotlin 1.9.24），由 Android Studio/命令行独立构建，不纳入 Go 构建链。

## 3. 架构与约定
- **版本号注入**：统一通过 ldflags `-X main.appVersion=${VERSION}` 注入运行时版本，CI 中 VERSION 取自 git tag，本地脚本取自 `git describe --tags`。
- **多架构策略**：Dockerfile 利用 `--platform=$BUILDPLATFORM` 在原生 CPU 上执行 Go 交叉编译（CGO_ENABLED=0），再产出多架构镜像，避免 QEMU 模拟编译带来的性能损耗；release CI 显式指定 `PLATFORMS: linux/amd64,linux/arm64`。
- **依赖隔离**：全部第三方包预置在 `vendor/`，构建命令强制 `-mod=vendor`，保证离线可复现；Go 标准库版本在 Dockerfile 注释中标注了安全提醒（1.22 EOL，建议升级至 1.26）。
- **插件打包**：server 镜像在构建期将 `plugins/` 目录 zip 为 `dist/plugins.zip`，随镜像分发，Agent 镜像直接 COPY plugins 目录以便 Python 插件运行。
- **健康检查**：server 镜像内置 `HEALTHCHECK` 探测 `/healthz`，compose 中 postgres 使用 `pg_isready` 作为依赖就绪条件。
- **数据持久化**：compose 将 PostgreSQL 数据写入具名卷 `aiops-pgdata`，VM 数据写 `./vm-data`，server 录制文件写 `./data`，均通过 volume 映射避免容器销毁丢失。

## 4. 开发者应遵循的规则
1. **新增 Go 依赖**：先 `go mod tidy`，再 `go mod vendor`，确保 `vendor/` 与 `go.mod` 同步提交。
2. **修改 Dockerfile**：若新增需要嵌入的资源，需在 builder 阶段 `COPY` 对应文件，并在最终镜像 stage 中 `COPY --from=builder` 到正确路径。
3. **版本号规范**：发布前打 `vX.Y.Z` 标签触发 release workflow；本地调试可用 `git tag -a v0.0.0-dev -m dev && git push origin v0.0.0-dev` 模拟。
4. **多架构构建**：本地测试多架构镜像需安装 QEMU 并使用 `docker buildx build --platform linux/amd64,linux/arm64 ...`；CI 已封装好，无需重复配置。
5. **Android 构建**：不在本仓库 Go 构建流程内，需单独使用 Android Studio 或 `./gradlew assembleRelease` 构建 APK。
6. **安全基线**：Dockerfile 注释明确标注 Go 1.22 存在 crypto/tls CVE，发布前应升级 `GO_VERSION` 并将新版 golang 镜像同步到 SWR；CodeQL 使用 1.26 构建，二者应保持对齐。
