# 时序数据与结构化数据分库存储

_来源：a8b268a → 791b7da 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
硬件数值指标（温度/功耗/风扇 RPM）和网络流量（bytes/s、packets/s）是高频时序数据，而硬件快照 JSON、事件记录、Flow 明细是低频结构化查询，两者访问模式完全不同。

## 决策驱动
- 查询性能
- 写入吞吐
- 运维复杂度

## 备选方案
- **全部存入 PostgreSQL** _（已否决）_ — 优点：单数据库运维简单、支持 JSONB 和事务；缺点：高时序写入压力大；范围聚合查询慢
- **全部存入 VictoriaMetrics** _（已否决）_ — 优点：时序查询极快、写入吞吐高；缺点：不支持 JSON 列和精确筛选导出
- **PostgreSQL（快照/事件/明细）+ VictoriaMetrics（数值指标/聚合）** — 优点：各司其职：PG 做结构化关联查询，VM 做时序聚合；VM 只存降采样指标减少体积；缺点：双存储运维成本

## 决策
Redfish 最新快照和事件存 PostgreSQL（JSONB + 索引），数值指标（温度/功耗/健康分数）写 VictoriaMetrics；NetFlow 聚合指标写 VM，可选的 Flow 明细存 PG 并带 7 天 TTL 清理。

## 影响
Server 层需维护两个写入路径；VM 指标名遵循 aiops_hardware_* / aiops_netflow_* 命名空间。后续可扩展告警规则同时消费两类数据源。
