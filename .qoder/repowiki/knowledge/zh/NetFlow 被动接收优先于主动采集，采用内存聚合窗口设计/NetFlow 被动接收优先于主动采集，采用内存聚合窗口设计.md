# NetFlow 被动接收优先于主动采集，采用内存聚合窗口设计

_来源：b8c1938 → a8b268a 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
网络流量采集需要支持高吞吐场景（默认 10000 flows/sec），且设备侧可能无法提供 SNMP/REST API。需要在保证性能的同时防止内存溢出。

## 决策驱动
- 高吞吐处理需求
- 内存上限保护
- 零 CGO 约束
- 向后兼容

## 备选方案
- **直接写入 VictoriaMetrics（无缓冲）** _（已否决）_ — 优点：实现简单、数据实时性高；缺点：VM 写入失败导致丢包、无法应对突发流量峰值
- **基于 Redis 的分布式队列** _（已否决）_ — 优点：持久化、可水平扩展；缺点：引入新依赖、增加延迟、违反零 CGO 约束
- **内存滑动窗口聚合 + 固定 UDP 缓冲区** — 优点：纯 Go 实现、内存可控、天然抗突发；缺点：进程重启丢失未 flush 数据、单机限制

## 决策
采用 net.ListenPacket 被动接收模式作为 P0，配合 5 分钟滑动窗口的 flowAggregator 内存聚合器，设置 100K flows 内存上限和 64KB UDP 缓冲区，超出时丢弃最小流量条目并记录 dropped_packets 计数。

## 影响
实现了高性能的流式聚合，但牺牲了进程重启时的数据持久性；背压机制保证了服务稳定性，但会丢失部分低流量记录。
